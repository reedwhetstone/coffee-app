## Findings by Severity

### P1 (should fix before merge)

- **Title:** Potential Race Condition in Secondary Fetch
- **Evidence:** `BeanProfileTabs.svelte` -> `handleCuppingSave` function.
- **Impact:** If the user saves cupping notes multiple times in rapid succession (or if the network is sluggish), the `refreshResponse` from an earlier save might resolve after a later save, potentially overwriting newer data with older data in the UI.
- **Correction:** While `processingUpdate` prevents multiple simultaneous saves from the UI, the secondary fetch should ideally be guarded or the `onUpdate` should check if the `selectedBean.id` still matches the `fullUpdatedBean.id` (though here it's the same bean, a version check or timestamp check would be safer). More importantly, ensure `processingUpdate` is set to `false` *only* after ALL async operations are complete (which it is), but consider using an AbortController for the refresh fetch if this component can be unmounted or switched during the fetch.

### P2 (important improvements)

- **Title:** Redundant code in fetch fallback
- **Evidence:** `BeanProfileTabs.svelte` lines 246-258.
- **Impact:** Logic duplication makes maintenance harder.
- **Correction:** Extract the `showCuppingForm = false; onUpdate(...)` logic into a small helper or simplify the nested `if/else` to avoid repeating the fallback logic three times.

```typescript
// Suggested simplification
let finalBean = null;
const refreshResponse = await fetch(`/api/beans?id=${selectedBean.id}`);
if (refreshResponse.ok) {

}

if (!finalBean) {
    finalBean = await response.json();
}

showCuppingForm = false;
onUpdate(finalBean);
```

## Assumptions Review

- Assumption: The `/api/beans?id=...` endpoint always returns the `coffee_catalog` join data.
- Validity: Valid
- Why: `src/routes/api/beans/+server.ts` uses `buildGreenCoffeeQuery` which explicitly selects `coffee_catalog!catalog_id (...)`.

- Assumption: `response.json()` on the PUT request returns the updated bean record.
- Validity: Valid
- Why: The PUT handler in `+server.ts` returns the result of `updateInventory`, which returns the updated row. However, the PUT response *does not* include the catalog join, which is why the PR added the secondary fetch.

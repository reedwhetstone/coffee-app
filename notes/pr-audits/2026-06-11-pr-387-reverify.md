# PR #387 Reverify: CLI-Owned Catalog Intelligence Patch

**Date:** 2026-06-11
**Reviewer:** OpenClaw subagent reverify
**Scope:** Focused re-verification after patch for the original supplier_list truthfulness findings.

## Verdict

`ready_with_fixes`

The patch resolves two of the three original findings cleanly: `non_wholesale_listings` is no longer emitted for unfiltered supplier aggregates, `avg_cup_score` is removed from the adapter contract and tool description, and the PR body now reflects the `@purveyors/cli ^0.20.1` bump plus the CLI-owned catalog-tool boundary.

One required truthfulness issue remains. `total_suppliers` and `truncated` are now computed from a new app-local `countMatchingSuppliers()` query capped by `fetchAllRows()` at `MAX_ROWS = 5000`. That means `supplier_list` can still undercount the supplier universe and report false non-truncation once more than 5,000 matching catalog rows exist. It also reintroduces a coffee-app-owned supplier filtering/counting helper even though the patch summary and ADR direction say supplier aggregate semantics should be CLI-owned.

## Original findings status

### Original P1: supplier_list aliases could report false coverage/counts

**Status:** Partially resolved; still needs patch.

Resolved pieces:

- `non_wholesale_listings` is only included when `non_wholesale_only` is active, where `supplier.total` truthfully means non-wholesale-filtered listings.
- `returned_suppliers` now separates returned rows from broader coverage.
- Limit-hit tests now assert `total_suppliers: 30`, `returned_suppliers: 25`, `truncated: true`, and a caveat for the generated 30-supplier case.

Remaining defect:

- `countMatchingSuppliers()` calls `fetchAllRows()`, and `fetchAllRows()` stops after `MAX_ROWS = 5000` rows without detecting whether additional pages exist.
- `@purveyors/cli`'s `supplierList()` in `node_modules/@purveyors/cli/dist/lib/catalog.js` paginates until a short page and returns `meta.rows_examined` for all matching rows. The app-local count does not match that behavior.
- Therefore, if CLI examines more than 5,000 matching rows, coffee-app can emit an undercounted `total_suppliers` and may leave `truncated` false because `reachedRequestedLimit` is computed from the undercount.
- The helper also ignores the CLI's `Unknown supplier` grouping for blank sources, creating another semantic mismatch.

This keeps the original truthfulness requirement from being fully satisfied.

## New regressions / concerns

### P1: `total_suppliers` / `truncated` can still be false above 5,000 matching rows

Files:

- `src/lib/services/marketTools.ts`

Evidence:

- `const MAX_ROWS = 5000;`
- `fetchAllRows()` loops while `offset < MAX_ROWS`, then returns without fetching one extra row or emitting truncation metadata.
- `countMatchingSuppliers()` uses `fetchAllRows()` to compute `total_suppliers`.
- `truncated` is computed from `response.meta.truncated || reachedRequestedLimit`; if the capped local count misses suppliers after row 5,000, this can still be false.

Impact:

The model-facing `supplier_list` tool description explicitly tells the agent to use `total_suppliers`, `returned_suppliers`, truncation, and caveats to describe coverage. A capped local count can make those coverage claims wrong.

Suggested fix:

Prefer one of these before merge:

1. Remove `total_suppliers` from the model-facing contract and rely on `returned_suppliers` plus CLI `rows_examined`/caveats until the CLI exposes a true supplier total.
2. Move true total/pre-limit supplier count into `@purveyors/cli` and consume that metadata here.
3. If coffee-app must keep a compatibility alias, use an uncapped helper with one-extra-row truncation detection and clear caveats when the count is incomplete. This is less aligned with ADR-006 than fixing/removing the field.

### P2: patch reintroduces app-local supplier semantics after saying they were removed

Files:

- `src/lib/services/marketTools.ts`
- `notes/decisions/006-cli-owned-portable-agent-tools.md`
- PR body in `pr.json`

Evidence:

- New `countMatchingSuppliers()` duplicates supplier filtering/counting outside the CLI.
- ADR-006 says reusable supplier aggregate semantics belong in `@purveyors/cli` and coffee-app should keep thin adapters only.
- PR body says the app removed remaining app-local supplier aggregation/filtering logic.

Impact:

This is not as severe as the count truthfulness issue, but it weakens the central architectural claim and doubles full-catalog supplier-list query work.
